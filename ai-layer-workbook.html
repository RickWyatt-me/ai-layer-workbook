The AI Layer Workbook

A step-by-step, plain-language guide to setting up Claude Code in a large codebase — built for people who can ship without being career engineers.

Most articles about getting AI to work in real codebases assume you already speak engineer. This one doesn't. It walks you through the same setup the best teams use — but with analogies, copy-paste templates, and prompts you hand to your own coding agent to do the heavy work for you.

Each phase has the same shape: why it matters, what you'll have at the end, concrete steps, copy-paste blocks, and a verify it worked section. Check items off as you go — your progress is saved in your browser.

How to use this workbook

  1. Read the three Orientation pages first. They give you the mental model — none of the implementation steps will land without them.
  2. Fill in the Personalize panel on the next page. The whole workbook adapts to your repo.
  3. Work through the nine Implementation phases in order. Each one takes 30 minutes to a couple of hours.
  4. Don't write CLAUDE.md files, hooks, or skills by hand. Use the prompts in this workbook — your own coding agent will do the work on your machine.
A note before you start

This workbook teaches the Claude Code conventions — CLAUDE.md, hooks, skills, plugins. But the work of setting them up can be done by whatever coding agent you already use. Use the picker in the top bar to set your agent; the wording adapts.

Concepts before code

Mental models

Seven analogies that let you reason about every piece of an AI Layer without knowing the technical names yet.

The AI Layer itself

Think of your codebase as a giant warehouse. Without help, every visitor — even an experienced one — wastes time finding the right aisle, asking where things are, and bumping into rules they didn't know existed.

The AI Layer is the signage, the floor map, the staff handbook, and the badge-access system — all bolted onto the warehouse so anyone walking in (human or AI) can get useful work done quickly without breaking things. The warehouse is your codebase. The AI Layer is what makes it navigable.

CLAUDE.md — the team handbook

A CLAUDE.md file is a short handbook for one part of your codebase. The handbook at the top of the warehouse covers company-wide rules: how we measure things, what departments exist. Handbooks in each department cover only that department's rules: how the billing team handles refunds, what the shipping team does with damaged goods.

The agent reads the top-level handbook first, then loads the relevant department handbook when it walks into that department. It never carries every handbook at once — that would be exhausting and most of them aren't relevant.

Hooks — automatic doors

Hooks are like automatic doors that fire when you cross a sensor. You don't ask them to open — they just do, at moments you've defined. A SessionStart hook is the door at the entrance ("welcome, here's today's floor map"). A Stop hook is the door at the exit ("what just happened on this visit — anything we should add to the handbook?"). They run on their own. You configure them once.

Skills — consultants on retainer

A skill is a specialist consultant who only shows up when their expertise is relevant. You wouldn't bring a tax attorney to every meeting; you'd call them when there's a tax question. Skills are scoped: the billing-specialist skill activates when the agent works in the billing department, and stays out of the way otherwise. This keeps the agent's "brain" — its context — from being cluttered with advice it doesn't need right now.

Subagents — scouts

A subagent is a scout you send ahead. Before your main party (the editing agent) descends into an unfamiliar part of the warehouse, you dispatch a scout — read-only, can't break anything, just observes — to map the area and report back. The scout returns with a clean report; the main agent then enters with the full picture instead of using up its own attention on discovery. Map first, then act.

LSP — GPS for code

The Language Server Protocol is GPS for your codebase. Without it, the agent searches for things by name (text matching) — like trying to find a specific "Main Street" by reading every street sign in every city. With it, the agent navigates by coordinates — actual definitions and references — so it finds the right thing the first time. This is the single highest-leverage upgrade for big codebases.

MCP — USB ports for Claude

An MCP server is a standard way to plug external tools into your agent. Think USB ports: any device that speaks USB can plug into any computer. Any tool that speaks MCP can plug into Claude Code. A search engine for your codebase, a connection to your ticket system, a custom analytics query — all become tools the agent can call.

Plugins — app installs

A plugin bundles a set of skills, hooks, and MCP servers into one installable package. Like installing an app on your phone: one tap, and everything comes with it pre-wired. When a teammate joins your project, they install your team's plugin and get the same setup you have — no manual copying of files.

Carry these analogies with you. Every implementation phase in this workbook maps back to one of them. If you ever get confused mid-phase, come back here.

Mechanics

How Claude Code actually works

No magic. Just a CLI tool that walks your filesystem and runs scripts at specific moments. Once you see the mechanics, the whole AI Layer stops feeling mysterious.

What "Claude Code" is, mechanically

Claude Code is a command-line program you install once on your computer. You open a terminal, navigate to your project folder, and type claude. That starts a session: an interactive conversation where you type requests and the agent reads files, runs commands, and edits code on your behalf — all locally, on your machine.

Unlike older AI tools that uploaded your entire codebase to a server to be "indexed," Claude Code does nothing of the sort. It works exactly like a careful new engineer would: opens files, greps for things, follows imports, looks at git history. The code never leaves your machine unless you explicitly send it somewhere.

What a "session" is

A session starts when you run claude and ends when you close that terminal or run /clear. Inside one session, the agent remembers everything you've discussed and every file it has looked at. Across sessions, it doesn't — each new session starts fresh.

This matters because the CLAUDE.md files are how the agent gets re-oriented at the start of every new session. Without them, every session starts blind.

What "the context" is, and why it matters

The context is the agent's short-term memory for the current session — a finite amount of text it can hold at once. Every file it reads, every output of every command, every line of conversation eats into that budget. When the budget gets low, the agent struggles.

This is why every design choice in the AI Layer is fundamentally about context economy: load only what's relevant right now, defer everything else until it's needed. CLAUDE.md hierarchy, path-scoped skills, read-only subagents — they all exist to spend the context budget wisely.

How hooks actually fire

A hook is just a command that Claude Code runs at a moment you've defined. You configure them in a file called .claude/settings.json. Here are the moments that exist:

Hook nameFires when…Typical use
SessionStartYou run claude or /clearInject orientation (active files, recent commits)
UserPromptSubmitYou hit Enter on a messageAdd context to your message before it's sent
PreToolUseBefore the agent runs a tool (e.g. Bash, Edit)Block dangerous commands; require confirmation
PostToolUseAfter a tool runsAuto-lint after every edit; log changes
StopThe agent finishes its current response to youSelf-improvement: reflect, propose CLAUDE.md updates
PreCompactBefore the agent compresses old conversationSnapshot state before it's summarised away

What a hook actually receives and returns

Each hook is just a program (a Python script, a shell command — anything). When the hook fires:

  1. Claude Code runs your program.
  2. It pipes a JSON payload to your program's standard input (the equivalent of typing into the program).
  3. Whatever your program writes to standard output (its console) gets injected back into the agent's context.
  4. Whatever your program writes to standard error shows up in the terminal for you to read.

That's it. Hooks aren't magic — they're scripts that run at lifecycle events and can speak to the agent through stdin/stdout. A bash one-liner is a valid hook. A 200-line Python program is also a valid hook. The simplest useful hook is just echo "Today is $(date)".

"Stop" doesn't mean what you'd think

"Stop" doesn't mean you closed the terminal. It means the agent finished its current response to your most recent message. The Stop hook fires every single turn. So whatever you put there must be cheap, or it must check whether anything actually changed before doing real work — which is exactly what the self-improvement pattern in Phase 6 does.

Skills, mechanically

A skill is a folder under .claude/skills/ containing a SKILL.md file. The first few lines of that file (the "frontmatter") declare a name, a description, and optionally a paths: field — a pattern matching where in your codebase this skill applies.

At every turn, Claude Code reads the descriptions of all available skills and decides which ones are relevant. Only relevant skills are loaded into context. A skill scoped to services/billing/** stays dormant until the agent works in services/billing.

Subagents, mechanically

A subagent is a second instance of Claude Code launched by the first. It has its own context window — its own short-term memory — separate from the parent. You define it in a single markdown file under .claude/agents/, including which tools it's allowed to use (e.g. only Read and Grep, no Edit or Write). When the parent dispatches the subagent, it does its work, returns a final report, and disappears. The parent receives only the report — not the subagent's reading list.

The whole picture, on one page

Here's what happens in a typical session, end to end:

  1. You type claude. SessionStart hook fires → prints orientation into context.
  2. Claude Code reads the root CLAUDE.md. As you work in subfolders, it loads those CLAUDE.mds too.
  3. You ask for something. Relevant skills auto-load based on which files you're touching.
  4. The agent reads files. For exact symbol lookups, it uses LSP. For structured search, it calls your MCP server.
  5. For a big unfamiliar area, the agent dispatches the explorer subagent first to map it.
  6. The agent edits files, runs your tests (filtered by the scoped-tests skill), reports back to you.
  7. Stop hook fires → notices what changed, spawns a background reflection that proposes CLAUDE.md updates while context is fresh.
  8. You type another request. Loop.

Every piece you build in this workbook hooks into one of those steps.

Phase 1

Foundation

Three files that give Claude Code a baseline: a high-level orientation, a list of what to ignore, and a list of what it's allowed to do.

Time: ~45 min Prereq: Claude Code installed Outcome: Three files in your repo root

Why this matters

Without these three files, every session starts with the agent doing the equivalent of asking "where am I, what do you want me to know, and what am I allowed to touch?" That's a context-burning waste. These files answer those questions once, version them in git, and travel with your repo.

What you'll have at the end

  • CLAUDE.md at your repo root — the lean orientation file
  • .claudeignore — generated files, lockfiles, build output excluded
  • .claude/settings.json — permissions and (later) hooks configured

Step 1 — Create the root CLAUDE.md

The single most important rule: this file is lean. It is not a manual. It is not a tutorial. It is repo-wide truths and the 3–5 gotchas that bite everyone, plus pointers to where more lives.

Open a terminal in your repo root and create the file:

Template — root CLAUDE.md (mobile app)

# your-repo

What this is in one line. (E.g. "Native mobile customer-support app — iOS and Android, shared backend.")

This root file is intentionally lean. It holds only what's true repo-wide.
Each platform and shared module has its own CLAUDE.md with local conventions;
Claude loads them automatically as it moves into that directory.

## Where things live

- `ios/` — Swift, Xcode project, iOS-specific code
- `android/` — Kotlin, Gradle, Android-specific code
- `shared/` — (if applicable) cross-platform contracts, protobufs, etc.
- `docs/` — engineering documentation

See each subdirectory's CLAUDE.md for the rules of that area.

## Critical gotchas (repo-wide)

- **Network contracts are shared.** API request/response shapes live in `shared/`
  (or whichever folder you use). A change there affects both platforms — run
  both test suites, not just one.
- **Versioning is per-platform.** iOS uses its own version scheme; Android uses
  its own. Don't conflate them.
- **Navigate by symbol, not by grep.** Both platforms have an LSP configured
  (sourcekit-lsp for Swift, kotlin-language-server for Kotlin). Prefer
  "go to definition" and "find references" over text search.
- **Tests scoped to platform.** A change in `ios/` runs the iOS test target.
  A change in `android/` runs the Android tests. A change in `shared/` runs both.

## Commands

```bash
# iOS
cd ios && xcodebuild test -scheme your-repo

# Android
cd android && ./gradlew test
```
Resist the urge to put everything in here

Conventions for the iOS code go in ios/CLAUDE.md (Phase 3). Conventions for the Android code go in android/CLAUDE.md. Skills go in skill files. If it doesn't apply repo-wide, it doesn't go in the root.

Step 2 — Create .claudeignore

This file tells Claude Code which paths to skip when searching or reading. It's the same idea as .gitignore — keep generated junk out of the agent's way.

Template — .claudeignore

# Generated artifacts and build output
build/
DerivedData/
.gradle/
.idea/
*.xcuserstate
*.xcodeproj/xcuserdata/

# Dependencies (let LSP/Xcode/Gradle manage these)
Pods/
node_modules/
.bundle/

# Caches
.swiftpm/
__pycache__/
*.pyc
.pytest_cache/

# Lockfiles — large, machine-generated
Podfile.lock
gradle.lockfile
package-lock.json
yarn.lock

# Local environment + secrets
.env
.env.*
*.pem
*.p12
*.keystore

# Mac/Windows clutter
.DS_Store
Thumbs.db

Step 3 — Create .claude/settings.json

This is where permissions and hooks live. For Phase 1, we'll set permissions only — hooks come in Phase 6.

Template — .claude/settings.json

{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "allow": [
      "Bash(xcodebuild test:*)",
      "Bash(./gradlew test:*)",
      "Bash(./gradlew build:*)",
      "Bash(git status:*)",
      "Bash(git diff:*)",
      "Bash(git log:*)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Read(./**/.env)",
      "Read(./**/.env.*)",
      "Read(./**/*.pem)",
      "Read(./**/*.p12)",
      "Read(./**/*.keystore)"
    ]
  }
}

The allow list pre-approves commands you'd normally have to confirm one at a time. The deny list blocks things outright — never accidentally rm -rf, never read secrets files.

Step 4 — Verify

  1. Open a terminal in your repo root.
  2. Run claude.
  3. Ask: "What's in your context right now? Just list the CLAUDE.md files you've loaded."
  4. You should see the root CLAUDE.md listed.

If the agent doesn't mention it, the file isn't being read — check the filename is exactly CLAUDE.md (case-sensitive) and that you're in the right directory.

05
Phase 2

Discovery — let your agent find the right spots

Don't guess where subdirectory CLAUDE.md files should go. Don't try to write them by hand. Have your agent audit your repo and tell you.

Time: ~30 min Prereq: Phase 1 complete Outcome: A prioritised list of where CLAUDE.md files belong

Why this matters

The right places for CLAUDE.md files aren't always obvious. Some folders are conventions-heavy; others are mostly boilerplate. Some have gotchas; others don't. Your agent — which has seen many codebases of similar shapes — can audit yours in minutes. Skipping this step usually results in either too many CLAUDE.mds (noise) or too few (the agent navigates blind in important areas).

What you'll have at the end

  • A markdown table from your agent: path, why this area benefits from a CLAUDE.md, priority (1–3)
  • A clear decision: which to do first, which to defer

Step 1 — Open your agent in your repo

In a terminal, navigate to your repo root and start Claude Code.

Step 2 — Paste this discovery prompt

This is the most important prompt in the workbook. It tells your agent not to write any files yet — only to audit and recommend.

Discovery prompt

Audit this codebase and recommend where CLAUDE.md files should go.

For each candidate directory:
1. Briefly explain what makes this area distinct (its domain, language, or conventions)
2. List the 2-3 gotchas an editor of this area absolutely needs to know
3. Note the scoped test/build command for this area, if there is one
4. Assign a priority: 1 (essential), 2 (valuable), 3 (nice to have)

Use signals like: directory name and structure, the languages present,
which folders import which other folders, the git log of recent changes,
and any existing documentation. Look at file counts to gauge complexity.

Output as a single markdown table with these columns:
Path | Domain/language | Gotchas | Scoped test command | Priority

Do NOT create any CLAUDE.md files. Just produce the table and a brief
paragraph at the end explaining your reasoning for the ordering.

Step 3 — Read the output critically

Your agent will produce a table. Don't accept it blindly. Read each row and ask yourself:

  • Is this area actually distinct? If two adjacent folders have identical conventions, they don't both need a CLAUDE.md.
  • Are the listed gotchas real? Or did the agent invent plausible-sounding ones?
  • Is the priority right for your day-to-day work? An area you edit twice a year is lower priority than one you touch weekly.

Push back. Ask follow-ups like "Why priority 1 for android/core but priority 2 for ios/Core? Aren't they parallel?" Your agent will adjust.

Step 4 — Pick your top 3–5 for Phase 3

Don't try to draft eight CLAUDE.md files in one go. Pick the highest-priority three to five and do those first. You can always add more later. Save the priority-3 candidates for the next iteration.

Common output for a mobile app

For a typical iOS + Android app, the audit usually surfaces: ios/ (Swift conventions), android/ (Kotlin/Gradle conventions), shared/ if you have one (network contracts), and possibly one networking module or one shared UI module per platform. Five files total is a good first pass.

Step 5 — Save the audit for reference

Copy the table from your agent's output into a file at docs/ai-layer/audit-2026.md (or wherever you keep engineering notes). You'll reference it in Phase 3 and again when you do your six-month review.

06
Phase 3

Subdirectory CLAUDE.md files

Use your agent to draft the per-area handbooks. Your job is to review and tighten — not to write from scratch.

Time: ~1.5 hrs (for 3–5 files) Prereq: Phase 2 audit complete Outcome: 3–5 working CLAUDE.md files in the right places

Why this matters

Subdirectory CLAUDE.md files are loaded additively as the agent walks into each area. They're where the local rules live — conventions specific to the iOS code, gotchas specific to the Android build, the test command for one service. Without them, the root CLAUDE.md balloons trying to cover everything, and the agent ignores rules that don't visibly apply.

What you'll have at the end

Three to five CLAUDE.md files committed to your repo at the paths your Phase 2 audit identified. Each one short, focused, and unmistakably about its area.

The rules every subdirectory CLAUDE.md must follow

  1. Under 60 lines. If it's longer, it's drifting into noise.
  2. Conventions, not tutorials. Don't explain what Swift is. Do explain how your Swift code handles error states.
  3. Gotchas first. The things that will bite an editor go above everything else.
  4. The scoped test/build command. Exactly what to run after editing this area.
  5. No duplication. If it's in the root CLAUDE.md, don't repeat it.

Step 1 — Draft each file using your agent

For each path from your Phase 2 audit, run this prompt:

Drafting prompt

Draft a CLAUDE.md for the directory `[PATH FROM AUDIT]`.

Follow these rules strictly:
- Total length: under 60 lines
- No facts that are obvious from the file extensions (don't say "this is
  written in Swift")
- No facts that already appear in the root CLAUDE.md
- Lead with 3-5 gotchas an editor of this area must know
- Include the exact scoped test/build command for this area
- Use bullet points, not paragraphs
- Reference other CLAUDE.md files where relevant rather than duplicating

Read the actual code in that directory before drafting. Specifically look
for: shared mutable state, error-handling patterns, naming conventions,
import boundaries (what this area imports vs what imports it), and any
unusual idioms.

Output only the CLAUDE.md content — no preamble, no commentary.

Step 2 — Review every line

Your agent will produce a draft. Read it line by line and ask:

  • Is this true? Agents occasionally hallucinate conventions. Check against the actual code.
  • Is this useful? "This module uses Swift's standard library" is not useful. Cut it.
  • Is this specific to this area? If it applies repo-wide, move it to the root.

A good rule: delete a line if removing it wouldn't surprise a future editor.

Example — what a good ios/CLAUDE.md looks like

# ios — Swift app target

## Conventions

- **Errors propagate via `Result<T, AppError>`** — never throw across module
  boundaries. `AppError` enum lives in `Core/AppError.swift`.
- **Networking goes through `APIClient`.** Direct `URLSession` calls are forbidden;
  they bypass auth headers and analytics.
- **No singletons except `APIClient` and `AnalyticsService`.** Everything else
  uses dependency injection through the app's `Resolver`.
- **State is async-let, not Combine.** We migrated off Combine in 2025;
  any remaining `@Published` properties are tech debt.

## Gotchas

- **Xcode's "Clean Build Folder" doesn't clear DerivedData.** If a build is
  acting strange, also `rm -rf ~/Library/Developer/Xcode/DerivedData/your-repo-*`.
- **The simulator caches keychain data.** Logout flows need an explicit
  `KeychainService.clear()` in tests, or the next test inherits the session.

## Tests

```bash
cd ios && xcodebuild test -scheme your-repo -destination 'platform=iOS Simulator,name=iPhone 15'
```

Step 3 — Verify each file loads

For each new CLAUDE.md, start a fresh session of your agent, ask it to read a file in that directory, and then ask: "What CLAUDE.md files do you have loaded right now?" Confirm the right ones appear.

Step 4 — Commit

Once each file is reviewed and verified, commit them as a single commit: git commit -m "AI Layer: add subdirectory CLAUDE.md files". This is the foundation everything else builds on.

Phase 4

LSP — symbol-level navigation

The single highest-leverage upgrade for large codebases. Stop the agent from greppping its way around. Give it actual coordinates.

Time: ~1 hr per language Prereq: Phase 1 complete Outcome: Symbol-level search and navigation working

Why this matters

If you have a function named fetch() in your codebase, grep will find it. Grep will also find the word fetch in 47 comments, 12 string literals, 8 unrelated method names, and the word prefetch three times. The agent then opens each file to figure out which one matters. That's a lot of wasted context for an answer the LSP gives you instantly.

For native mobile development, this is especially valuable because Swift and Kotlin are strongly typed — the LSP has rich type information and can resolve symbols precisely.

What you'll have at the end

Two language servers running locally (one per platform), wired into Claude Code via a plugin. Your agent now has "go to definition" and "find all references" as native capabilities.

iOS — Swift via sourcekit-lsp

Step 1: Confirm Xcode is installed

sourcekit-lsp ships with Xcode. Run:

xcrun --find sourcekit-lsp

You should see a path like /Applications/Xcode.app/Contents/Developer/usr/bin/sourcekit-lsp. If you get an error, install Xcode from the App Store, then run sudo xcode-select --switch /Applications/Xcode.app/Contents/Developer.

Step 2: Install the Claude Code code-intelligence plugin

In your terminal, in your repo root:

claude
/plugin install code-intelligence

Step 3: Configure it for Swift

Ask your agent: "Configure the code-intelligence plugin to use sourcekit-lsp for Swift files in ios/. Update .claude/settings.json if needed."

Step 4: Verify

Open a Swift file. Ask the agent: "Find all references to APIClient in the iOS code." If you see file:line references and the agent reports the count cleanly without opening 30 files first, the LSP is working.

Android — Kotlin via kotlin-language-server

Step 1: Install kotlin-language-server

On macOS via Homebrew:

brew install kotlin-language-server

On Linux or Windows, follow the official install guide.

Step 2: Verify the binary

which kotlin-language-server

You should see a path.

Step 3: Wire into Claude Code

Ask your agent: "Configure the code-intelligence plugin to use kotlin-language-server for Kotlin files in android/. Update .claude/settings.json if needed."

Step 4: Verify

Open a Kotlin file. Ask the agent: "Find all references to the Repository interface in the Android code." Same test as iOS — clean file:line references, no scattered file-reading.

What it feels like when LSP is working

You'll notice:

  • Refactor questions ("rename this function across the codebase") get answered in seconds, not minutes.
  • "Where is this defined?" gets one answer, not three guesses.
  • The agent's responses become more confident — because it's working from real type information, not pattern-matching.

Common stumbling blocks

"sourcekit-lsp can't find my Swift package / Xcode project"

sourcekit-lsp needs to know the build context. Open the project in Xcode once first so it generates the DerivedData, then restart Claude Code. For Swift packages, ensure Package.swift is at the root of the indexed area.

"kotlin-language-server is slow on first start"

Expected. It builds an index of your Kotlin code on first use. Give it 5–10 minutes on a large codebase. Subsequent starts are fast.

"It works for one platform but not the other"

You probably configured the plugin for only one language. Run the configuration step for both.

Phase 5

Skills — reusable expertise on demand

Capture the things your team always has to re-explain. Skills load only when relevant, so your agent's "brain" stays uncluttered.

Time: ~30 min per skill Prereq: Phase 3 complete Outcome: 3–5 skills covering your most frequent workflows

Why this matters

Some pieces of expertise apply only sometimes. Money-handling rules apply only when working in billing. Test patterns apply only when writing tests. If you put all of it in CLAUDE.md, every session carries it — bloating the context for no reason. Skills sit dormant until they're needed.

What you'll have at the end

A .claude/skills/ folder with three to five skills, each scoped via the paths: field so it auto-loads only in the relevant part of your codebase.

The skills to build first

For a native mobile app, the highest-leverage starting set:

SkillScopeWhat it captures
scoped-testsrepo-wideWhich test command to run for each kind of change
ios-add-screenios/**The exact steps to add a new screen in your iOS app
android-add-screenandroid/**Same, for Android
network-contract-changeshared/** or whereverSteps for changing an API contract — both platforms need updating

Anatomy of a skill

A skill is a folder with a SKILL.md file. The first few lines (the "frontmatter") declare the skill's name, when to use it, and where it applies. The body is the actual content.

Template — .claude/skills/scoped-tests/SKILL.md

---
name: scoped-tests
description: >-
  Use after changing code, before claiming work is done — picks the
  correctly scoped test command instead of running the full suite.
paths:
  - ios/**
  - android/**
  - shared/**
---

# Scoped test runner

Running the whole suite on a one-platform change wastes time and context.
Pick the narrowest command that still covers the change.

| What you changed | Run |
|------------------|-----|
| `ios/**` | `cd ios && xcodebuild test -scheme your-repo` |
| `android/**` | `cd android && ./gradlew test` |
| `shared/**` | Both — `shared` is consumed by both platforms |

**Rule:** if `shared/` changed, always run both test suites. A breaking
change there ships to production if either platform isn't checked.

Template — .claude/skills/ios-add-screen/SKILL.md

---
name: ios-add-screen
description: >-
  Use when adding a new screen to the iOS app. Walks the exact steps so
  the new screen follows your-repo's navigation, state, and analytics conventions.
paths: ios/**
---

# Adding an iOS screen

Activates when working in `ios/`. Follow these steps in order.

1. **Create the View** in `Features/<FeatureName>/` as a SwiftUI `View`.
   Use the project's `ViewModel` protocol — no logic inside the view.
2. **Create the ViewModel** in the same folder. Inherit from
   `BaseViewModel<State, Action>`. State is an enum, actions are an enum.
3. **Register the route** in `Navigation/AppRoutes.swift`. A screen not
   registered there cannot be navigated to.
4. **Add the analytics event** in `Analytics/Events.swift`.
   Convention: `screen_viewed_<feature_name>`.
5. **Add a snapshot test** in `FeatureName/Tests/<FeatureName>SnapshotTests.swift`.

Full checklist with code patterns: [references/screen-checklist.md](references/screen-checklist.md).

The "paths:" field — pay attention to this

The paths: field is what makes skills lightweight. A skill scoped to ios/** doesn't even register in the agent's awareness when you're working in Android. This is how you can have a dozen skills and only the relevant 1–2 load at any time.

  • ios/** — anywhere under ios/ at any depth
  • ios/Features/** — only under the Features folder
  • Multiple paths via a YAML list (see scoped-tests example above)

Progressive disclosure — keep skills lean

Your skill's SKILL.md should be short — under 50 lines. For longer guidance (full code patterns, edge cases), create a references/ folder beside it and link to specific files. The agent loads the reference only when it needs the detail.

Step 1 — Use your agent to draft each skill

For each skill in your starting set, run this prompt:

Skill-drafting prompt

Draft a skill at `.claude/skills/[NAME]/SKILL.md` for the following task:

  [DESCRIBE THE TASK — e.g. "Adding a new screen to our iOS app"]

Follow these rules:
- Use the YAML frontmatter format (name, description, paths)
- The description must clearly state when to use this skill
- The paths: field should restrict it to the relevant directories
- Keep SKILL.md under 50 lines
- Use numbered steps for procedures, tables for reference data
- For any deep detail, create a stub at references/[name].md and link to it

Read [example file or area] before drafting so the skill reflects how the
actual code is structured.

Output the SKILL.md content only.

Step 2 — Test that the skill loads

After creating a skill, start a fresh session, open a file in the skill's scope, and ask: "What skills do you have loaded right now?" The skill should appear. Then ask it a question the skill addresses; you should see the skill's guidance reflected in the answer.

Step 3 — Verify scoping works

Switch to a file outside the skill's scope and ask the same question. The skill should not be loaded; the answer should be more generic. If it loads anyway, your paths: field is too broad.

Phase 6

Hooks — automation that runs itself

Two hooks. The first gives every session instant orientation. The second makes your AI Layer improve itself.

Time: ~2 hrs Prereq: Phase 5 complete Outcome: SessionStart + Stop hooks running, self-improvement loop active

Why this matters

You can write the best CLAUDE.md files in the world, but they drift. Code evolves; the documentation about it lags. Hooks are how you fight drift automatically. The SessionStart hook ensures every session starts oriented to now (not last week). The Stop hook reflects on what just happened and proposes documentation updates while context is fresh.

What you'll have at the end

  • A SessionStart hook that injects today's "active areas" and recent commits at every session start
  • A Stop hook that detects changed areas and spawns a background reflection that proposes CLAUDE.md edits
  • A new file .claude/claude-md-review.md that accumulates proposed updates for you to review

Start simple — SessionStart only

Don't try to build both hooks at once. Get SessionStart working first; it's the simpler one and gives you immediate value.

Template — .claude/hooks/session_start_context.py

"""SessionStart hook — dynamic per-area orientation.

Prints a short orientation block at the start of every Claude Code session.
Claude Code injects this stdout into the session context, so Claude starts
already knowing which area has active work — and the recent direction of
travel from git history — without spending a turn re-exploring.

Tested standalone: `python .claude/hooks/session_start_context.py`
"""
from __future__ import annotations

import os
import subprocess
import sys
from pathlib import Path

_EXCLUDE_DIRS = frozenset({
    ".git", ".venv", "venv", "env", "node_modules", "__pycache__",
    ".pytest_cache", ".gradle", "build", "DerivedData", "Pods",
})


def _project_root() -> Path:
    project = os.environ.get("CLAUDE_PROJECT_DIR")
    return Path(project) if project else Path(__file__).resolve().parents[2]


def _claude_md_areas(root: Path) -> set[str]:
    areas: set[str] = set()
    for dirpath, dirnames, filenames in os.walk(root):
        dirnames[:] = [d for d in dirnames if d not in _EXCLUDE_DIRS]
        if "CLAUDE.md" in filenames:
            rel = Path(dirpath).relative_to(root).as_posix()
            if rel != ".":
                areas.add(rel)
    return areas


def _area_of(changed: str, areas: set[str]) -> str | None:
    parts = changed.split("/")
    for depth in range(len(parts) - 1, 0, -1):
        candidate = "/".join(parts[:depth])
        if candidate in areas:
            return candidate
    return None


def _working_tree_changes(root: Path) -> list[str]:
    try:
        result = subprocess.run(
            ["git", "status", "--porcelain"],
            cwd=root, capture_output=True, text=True, timeout=5,
        )
    except (OSError, subprocess.SubprocessError):
        return []
    return [line[3:].strip().replace("\\", "/")
            for line in result.stdout.splitlines() if len(line) > 3]


def _recent_commits(root: Path, limit: int = 5) -> list[str]:
    try:
        result = subprocess.run(
            ["git", "log", f"-{limit}", "--pretty=format:%h %s"],
            cwd=root, capture_output=True, text=True, timeout=5,
        )
    except (OSError, subprocess.SubprocessError):
        return []
    return [line.strip() for line in result.stdout.splitlines() if line.strip()]


def main() -> None:
    try: sys.stdin.read()
    except: pass

    root = _project_root()
    governed = _claude_md_areas(root)
    changes = _working_tree_changes(root)
    active = sorted({_area_of(p, governed) for p in changes if _area_of(p, governed)})

    lines = ["## Session orientation", ""]
    if active:
        lines.append(f"Active area(s) this session: **{', '.join(active)}**.")
        lines.append("Load the matching CLAUDE.md in each before editing.")
    else:
        lines.append("Working tree is clean — no area has pending work.")

    commits = _recent_commits(root)
    if commits:
        lines.append("")
        lines.append("Recent commits (newest first):")
        lines.extend(f"- {commit}" for commit in commits)

    print("\n".join(lines))


if __name__ == "__main__":
    main()

Then add the self-improving Stop hook

This is the more sophisticated piece. It's split into two scripts because the work is split: a cheap trigger that runs every turn, and an expensive reflection that runs in the background only when something actually changed.

Show the full Stop hook script (propose_claude_md.py)

This file lives at .claude/hooks/propose_claude_md.py. It's adapted from Cole's helpline repo.

"""Stop hook trigger — cheap, deterministic part of the self-improving loop.

Detects which CLAUDE.md-governed areas changed, dedups against prior runs,
and spawns reflect_claude_md.py in the background. Returns immediately so
the turn never blocks on an LLM call.

Three guards: recursion lock, diff fingerprint dedup, deterministic fallback
if the reflector fails.
"""
from __future__ import annotations

import hashlib, os, subprocess, sys
from pathlib import Path

_EXCLUDE_DIRS = frozenset({
    ".git", ".venv", "venv", "env", "node_modules", "__pycache__",
    ".pytest_cache", ".gradle", "build", "DerivedData", "Pods",
})
_LOCK_ENV = "AILAYER_REFLECT_LOCK"
_STATE_FILE = ".claude/.claude-md-review-state"
_REFLECTOR = "reflect_claude_md.py"
_DETACHED_PROCESS = 0x00000008


def _project_root() -> Path:
    project = os.environ.get("CLAUDE_PROJECT_DIR")
    return Path(project) if project else Path(__file__).resolve().parents[2]


def _git(args, root, timeout=5):
    try:
        return subprocess.run(["git", *args], cwd=root, capture_output=True,
                              text=True, timeout=timeout).stdout
    except (OSError, subprocess.SubprocessError):
        return ""


def _claude_md_areas(root):
    areas = set()
    for dp, dns, fns in os.walk(root):
        dns[:] = [d for d in dns if d not in _EXCLUDE_DIRS]
        if "CLAUDE.md" in fns:
            rel = Path(dp).relative_to(root).as_posix()
            if rel != ".": areas.add(rel)
    return areas


def _area_of(changed, areas):
    parts = changed.split("/")
    for depth in range(len(parts) - 1, 0, -1):
        c = "/".join(parts[:depth])
        if c in areas: return c
    return None


def _touched_areas(root):
    governed = _claude_md_areas(root)
    out = set()
    for line in _git(["status", "--porcelain"], root).splitlines():
        if len(line) <= 3: continue
        path = line[3:].strip().replace("\\", "/")
        a = _area_of(path, governed)
        if a: out.add(a)
    return out


def _spawn_reflector(reflector, root):
    flags = 0; new_session = False
    if os.name == "nt":
        flags = subprocess.CREATE_NEW_PROCESS_GROUP | _DETACHED_PROCESS
    else:
        new_session = True
    try:
        subprocess.Popen(
            [sys.executable, str(reflector)],
            cwd=str(root),
            stdin=subprocess.DEVNULL, stdout=subprocess.DEVNULL, stderr=subprocess.DEVNULL,
            creationflags=flags, start_new_session=new_session,
        )
    except Exception:
        return False
    return True


def main():
    try: sys.stdin.read()
    except: pass
    if os.environ.get(_LOCK_ENV): return 0

    root = _project_root()
    areas = _touched_areas(root)
    if not areas: return 0

    fp = hashlib.sha256(_git(["diff", "HEAD", "--", *sorted(areas)], root)
                        .encode("utf-8", "replace")).hexdigest()
    state = root / _STATE_FILE
    try:
        if state.read_text().strip() == fp: return 0
    except OSError: pass

    reflector = Path(__file__).with_name(_REFLECTOR)
    if not reflector.is_file(): return 0
    if not _spawn_reflector(reflector, root): return 0

    try:
        state.parent.mkdir(parents=True, exist_ok=True)
        state.write_text(fp)
    except OSError: pass

    print(f"[ai-layer] {len(areas)} area(s) changed — reflecting in background", file=sys.stderr)
    return 0


if __name__ == "__main__":
    sys.exit(main())
Show the reflector script (reflect_claude_md.py)

This file lives at .claude/hooks/reflect_claude_md.py. It calls headless claude -p to do the actual reflection.

"""The reasoning half of the self-improving Stop hook.

Gathers the session's working-tree diff plus the current CLAUDE.md of every
area that changed, asks Claude (headless `claude -p`) to judge whether those
conventions still hold, and writes the proposal to .claude/claude-md-review.md.

If `claude` is unavailable, writes a deterministic "re-check these files"
note instead.
"""
from __future__ import annotations

import os, shutil, subprocess, sys
from datetime import datetime
from pathlib import Path

_REVIEW = ".claude/claude-md-review.md"
_LOCK = "AILAYER_REFLECT_LOCK"
_MAX_DIFF = 12_000
_TIMEOUT = 180


def _root():
    p = os.environ.get("CLAUDE_PROJECT_DIR")
    return Path(p) if p else Path(__file__).resolve().parents[2]


def _git(args, root):
    try:
        return subprocess.run(["git", *args], cwd=root, capture_output=True,
                              text=True, timeout=10).stdout
    except: return ""


def _touched(root):
    EXCLUDE = {".git", ".venv", "node_modules", "__pycache__", ".gradle",
               "build", "DerivedData", "Pods"}
    areas = set()
    for dp, dns, fns in os.walk(root):
        dns[:] = [d for d in dns if d not in EXCLUDE]
        if "CLAUDE.md" in fns:
            rel = Path(dp).relative_to(root).as_posix()
            if rel != ".": areas.add(rel)
    touched = {}
    for line in _git(["status", "--porcelain"], root).splitlines():
        if len(line) <= 3: continue
        path = line[3:].strip().replace("\\", "/")
        for depth in range(len(path.split("/")) - 1, 0, -1):
            c = "/".join(path.split("/")[:depth])
            if c in areas:
                touched[c] = touched.get(c, 0) + 1
                break
    return touched


def _build_prompt(root, areas, diff):
    blocks = []
    for area in sorted(areas):
        md = root / area / "CLAUDE.md"
        content = md.read_text() if md.is_file() else "(no CLAUDE.md yet)"
        blocks.append(f"### {area}/CLAUDE.md\n\n{content}")
    return f"""You are auditing whether a codebase's CLAUDE.md files still match \
reality after a coding session.

Below is the git diff of uncommitted changes, then each touched area's CLAUDE.md.

For EACH area, output one of:
- `No change needed` — the CLAUDE.md still holds
- A concrete proposed edit: the specific line(s) to add, change, or remove,
  plus one sentence on why

Only propose updates for genuine new conventions, gotchas, or constraints.
Skip stylistic rewrites. Be terse. No tools needed.

## Git diff
```
{diff}
```

## Current CLAUDE.md files
{chr(10).join(blocks)}
"""


def _run_claude(prompt, root):
    claude = shutil.which("claude")
    if not claude: return None
    env = dict(os.environ); env[_LOCK] = "1"
    try:
        r = subprocess.run(
            [claude, "-p", "--output-format", "text"],
            cwd=root, input=prompt, capture_output=True, text=True,
            timeout=_TIMEOUT, env=env,
        )
    except: return None
    return r.stdout.strip() if r.returncode == 0 else None


def _fallback(areas, stamp):
    lines = [f"# CLAUDE.md review — {stamp}", "",
             "_`claude` CLI unavailable. Re-check these areas by hand._", ""]
    for area, n in sorted(areas.items()):
        lines.append(f"- **{area}** ({n} file(s))")
    return "\n".join(lines) + "\n"


def reflect():
    if os.environ.get(_LOCK): return 0
    root = _root()
    areas = _touched(root)
    if not areas: return 0
    diff = _git(["diff", "HEAD", "--", *sorted(areas)], root)
    if len(diff) > _MAX_DIFF:
        diff = diff[:_MAX_DIFF] + "\n... (truncated)"
    stamp = datetime.now().isoformat(timespec="seconds")
    out = _run_claude(_build_prompt(root, areas, diff), root) if diff.strip() else None
    body = (f"# CLAUDE.md review — {stamp}\n\n{out}\n"
            if out else _fallback(areas, stamp))
    review = root / _REVIEW
    try:
        review.parent.mkdir(parents=True, exist_ok=True)
        review.write_text(body)
    except: return 1
    return 0


if __name__ == "__main__":
    sys.exit(reflect())

Phase 7

Subagent — the read-only scout

Send a scout ahead. Map the territory cheaply. Then act with the full picture.

Time: ~20 min Prereq: Phase 3 complete Outcome: A reusable explorer subagent

Why this matters

When your main agent enters an unfamiliar subsystem, it spends a lot of attention on discovery before any actual editing. By the time it understands the area, half its context budget is gone. A subagent solves this: it explores with its own context, returns only a summary, and the main agent edits with that summary in hand — fresh context, full picture.

The trick is that the subagent must be read-only. If it can edit, it might start trying to "fix" things while exploring, defeating the purpose. We enforce read-only by not giving it write tools — not by asking nicely.

What you'll have at the end

A single file at .claude/agents/explorer.md that defines a reusable scout. Any time you need to map a subsystem before editing it, you dispatch the explorer.

Step 1 — Create the explorer

The explorer is defined in one markdown file. Frontmatter sets its name, scope, and — critically — which tools it's allowed to use.

Template — .claude/agents/explorer.md

---
name: explorer
description: >-
  Read-only subsystem explorer. Use it to map an unfamiliar service or
  module BEFORE editing — it explores with its own context window and
  reports back, so the main agent edits with the full picture instead
  of burning its context on discovery.
tools: Read, Grep, Glob
model: sonnet
---

# Explorer subagent

You map one subsystem of the your-repo codebase. You are **genuinely
read-only**: your only tools are `Read`, `Grep`, and `Glob`. There is no
`Write` or `Edit`, so you *cannot* modify the codebase even if asked.

## When you are invoked

You will be given one subsystem to map — a top-level folder like `ios/`,
`android/`, or a submodule like `ios/Features/Billing`.

## What to do

1. Read that subsystem's `CLAUDE.md` first if it has one.
2. Use Glob and Grep to find: entry points, public types/classes, what it
   imports from other modules, and what imports it.
3. Identify the gotchas — shared state, error contracts, anything surprising.
4. Return your findings as your final report, under these headings:
   - **Entry points** — where work starts
   - **Key types & functions** — the public surface
   - **Dependencies** — what it imports, what imports it
   - **Gotchas** — what would bite an editor
   - **Suggested fixes** — anything that looks wrong; *describe* it,
     since you cannot apply it

## Why read-only

Running exploration and editing in one session spends the editing context
on discovery. A separate read-only explorer keeps them apart. Having no
write tools is the *guarantee* of that separation, not a polite request
you could break.

Step 2 — Dispatch it

In a Claude Code session, ask: "Use the explorer subagent to map android/core and report back."

The explorer will run in a separate context window, do its work, and return a report. Your main agent receives the report — not the explorer's reading list — and can then edit with a complete picture.

When to use the explorer

  • You're about to refactor code you haven't touched in months.
  • You're picking up a service or module a teammate built.
  • You're debugging an issue that crosses several modules.
  • You're estimating the scope of a feature that touches an unfamiliar area.

When not to use it: a quick edit to a file you know well. The dispatch overhead isn't worth it for small, scoped changes.

Phase 8 · Advanced

MCP — structured tools for your agent

For when grep noise becomes a real problem, or you want to plug your agent into something only your team has.

Time: half a day or more Prereq: Phases 1–7 done and bedded in Outcome: Optional. Skip unless you need it.

Why this matters — and when it doesn't

An MCP server is a custom tool your agent can call. The most common uses for a custom MCP:

  • Codebase search by AST instead of grep. Useful for million-line codebases or codebases where method names are common words.
  • Internal documentation lookup across systems your team uses (Notion, internal wikis, design docs).
  • Ticket/issue queries against Jira, Linear, Shortcut, etc.
  • Deployment or analytics queries for your specific platforms.

You probably don't need this in your first month. The patterns from Phases 1–7 deliver most of the value. MCP servers are a high-investment, high-ceiling addition you make when you've outgrown the basics.

What an MCP server is, in one paragraph

An MCP server is a small program (Python, Node, anything) that exposes a set of "tools" — functions the agent can call. It runs locally and communicates over standard input/output. You wire it into Claude Code via a file called .mcp.json. From the agent's point of view, calling your MCP server's tool feels the same as calling any built-in tool.

The high-value first MCP — AST-based codebase search

The single most useful custom MCP for a large mobile codebase is a structured search server. Instead of grep returning 200 matches for "send," your AST-based search returns "the 3 actual function definitions named send in your repo, with their qualified names and line numbers."

Cole's helpline repo includes a reference implementation in Python. For Swift and Kotlin you'd write something similar using each language's AST tooling (SwiftSyntax and Kotlin's PSI, respectively) — or use the LSP from Phase 4, which already provides much of this.

When the LSP is enough vs when you need MCP

SituationLSP enough?MCP useful?
"Where is Foo defined?"YesNo
"Find all callers of Foo"YesNo
"List every API endpoint our app calls"No — that's a custom queryYes
"Show me every screen using OldDeprecatedView"YesNo
"Query our internal ticketing system"NoYes
"What changed across services in the last release?"NoYes

Step 0 — Decide if you need this

Before investing days into a custom MCP, ask: "After three months with Phases 1–7, what queries am I still hand-running because my agent can't?" Those are your candidate MCP tools. If the answer is "none," skip this phase.

Step 1 — If you decide yes, start with one well-defined tool

Don't try to build a Swiss army knife MCP. Pick one query that you run by hand more than weekly. Build an MCP server with that one tool. Live with it for a sprint. Add more tools only when you've felt the pull.

What's in this phase, deliberately

This workbook deliberately doesn't include a full MCP server template — because the right design depends entirely on what you're querying. The prompt below gets your agent to scaffold one for you.

MCP scaffolding prompt

I want to build a custom MCP server for my repo. The tool it should expose:

  [DESCRIBE — e.g. "Given a feature name, return every Swift file
   that references it and every Kotlin file that references it,
   so I can audit cross-platform consistency."]

Scaffold:
1. The Python (or Node) script for the MCP server
2. The .mcp.json entry to wire it in
3. A README explaining how to run and test it standalone

Use the MCP SDK conventions (the `mcp` Python package or `@modelcontextprotocol/sdk`
for Node). The tool's docstring must be precise — the agent decides when to call it
based on that description.

Output: the files, with the path each should live at.

Phase 9

Plugin — distribute what works

Once your AI Layer earns its keep, bundle the portable pieces so teammates (or a future project) get them in one command.

Time: ~2 hrs Prereq: Phases 1–7 working Outcome: An installable plugin teammates can adopt with /plugin install

Why this matters

Without a plugin, knowledge stays tribal. The first engineer to set up their AI Layer reaps the reward; everyone else either copies files by hand or stays without. A plugin turns your setup into a one-command install for anyone on your team — or in your community.

What travels vs what stays

This is the crucial distinction. Some of your AI Layer is repo-specific (the CLAUDE.md files describing your conventions). Some of it is portable (the hooks that detect changed areas based on CLAUDE.md presence — they work in any repo). The plugin bundles only the portable pieces.

Stays in your repoGoes in the plugin
Root CLAUDE.mdThe Stop hook (works in any repo with a CLAUDE.md hierarchy)
Subdirectory CLAUDE.md filesThe reflector script
Domain-specific skills (ios-add-screen, etc.)Generic skills (scoped-tests)
SessionStart hook (references your structure)The explorer subagent
.claudeignore (your file patterns)Any custom MCP servers

Step 1 — Decide what's portable

Walk through your .claude/ folder. For each item, ask: "Would this work in a completely different repo, with a different structure?"

  • If yes → portable, goes in the plugin
  • If no → stays in this repo's .claude/

Step 2 — Set up the plugin folder

Create a separate folder — could be inside your repo (e.g. tooling/your-team-ai-layer/) or in a completely separate repo (better for sharing across projects).

your-repo-ai-layer/
└── tooling/
    ├── .claude-plugin/
    │   └── marketplace.json     ← the marketplace manifest
    └── your-team-ai-layer/
        ├── .claude-plugin/
        │   └── plugin.json       ← the plugin manifest
        ├── hooks/
        │   ├── hooks.json
        │   ├── propose_claude_md.py
        │   └── reflect_claude_md.py
        ├── agents/
        │   └── explorer.md
        ├── skills/
        │   └── scoped-tests/
        │       └── SKILL.md
        ├── mcp/                  ← if you have an MCP server
        │   └── server.py
        └── README.md

  "version": "0.1.0",

    }

Step 3 — Make hooks/agents/skills portable

Anything in the plugin needs to reference its own directory via ${CLAUDE_PLUGIN_ROOT} (the plugin's own path when installed). Update the hooks.json:

{
  "hooks": {
    "Stop": [
      { "hooks": [{ "type": "command",
        "command": "python \"${CLAUDE_PLUGIN_ROOT}/hooks/propose_claude_md.py\"" }] }
    ]
  }
}

Step 4 — Install & test

From another repo entirely (or after temporarily removing your .claude/hooks/):

claude
/plugin marketplace add /path/to/your-repo-ai-layer/tooling
/plugin install your-team-ai-layer@your-team-tooling

Verify the hook fires, the subagent is dispatchable, and the skill loads in the new repo. If any of those don't work, the most common cause is a path that references the original repo's structure — fix to use ${CLAUDE_PLUGIN_ROOT}.

Step 5 — Share

Once it works, push the plugin to a git repo your teammates can clone. They run the two commands above against their clone, and they're set up.

Maintenance

An AI Layer set up once and forgotten degrades. A small recurring review keeps it sharp.

The 3–6 month review cadence

Every three to six months, schedule an hour to walk through your AI Layer. The trigger for an off-cycle review: a major new model release, or a noticeable drop in agent quality.

What to look at, in order

1. The accumulated .claude/claude-md-review.md

This file collects every proposed CLAUDE.md update from your Stop hook. Read through it. Apply the genuinely useful ones; ignore the noise; clear the file.

2. CLAUDE.md drift

Open each CLAUDE.md (root and subdirectory). Ask:

  • Do the gotchas still describe real gotchas? (Were any fixed?)
  • Are the conventions still followed? (Or has the code drifted?)
  • Are the test/build commands still correct?

Delete what's no longer true.

3. Skills that exist to compensate for old model limits

This one is subtle. Some skills exist because earlier models needed hand-holding ("always break refactors into single-file changes"). Newer models often handle those scenarios natively. Skills built to compensate for model weaknesses can become a drag once those weaknesses are gone.

Audit each skill: is the limitation it works around still real? If not, delete the skill.

4. Hooks that intercept what the model now does natively

Same principle as skills. If you wrote a hook to enforce something the model used to skip, check whether the model still skips it. If not, the hook is overhead.

5. New skills worth adding

What workflows have you found yourself explaining repeatedly over the last quarter? Those are skill candidates. Run the skill-drafting prompt from Phase 5.

The hardest discipline: deleting

It feels productive to add things. It feels less productive to delete. But your AI Layer's quality is shaped by what's not in it as much as what is. Be ruthless.

Re-verifying after major model updates

When Anthropic ships a major Claude model release, do a focused review:

  1. Test your most common workflows against the new model with your current AI Layer.
  2. Note where the new model behaves differently — better or worse.
  3. Update CLAUDE.md, skills, and hooks accordingly.
  4. Document what you changed and why (a short note in your repo).

Glossary

Every technical term, in plain language.

Agent
A program that uses an AI model to read, write, and run things on your behalf. Claude Code, Codex, Cursor, and Cline are all agents.
AI Layer
The bundle of configuration and tooling — CLAUDE.md files, hooks, skills, subagents, LSP, MCP, plugins — that makes a codebase navigable for an AI agent. Cole Medin's term; the article calls it "the harness."
AST
Abstract Syntax Tree. A structured representation of code that knows the difference between a function definition and a comment that just mentions the function's name. Powers precise code search.
CLAUDE.md
A markdown file Claude Code reads automatically. A root file holds repo-wide truths; subdirectory files hold local conventions.
CLI
Command-Line Interface. A program you run by typing commands in a terminal. Claude Code is a CLI tool.
Context (or context window)
The agent's short-term memory for the current session. Finite. Every file read, every command output, every line of conversation consumes some of it.
Frontmatter
The block at the top of a markdown file between two --- lines, holding structured metadata (name, description, paths). Used in skills and subagents.
Grep
A text-search command. Fast and useful, but finds the search term anywhere — in code, in comments, in strings — without knowing the difference.
Hook
A script Claude Code runs automatically at a specific lifecycle moment (session start, after a tool call, etc.).
LSP
Language Server Protocol. A system that gives an editor (or an agent) symbol-level understanding of code — go-to-definition, find-references, type information.
MCP
Model Context Protocol. A standard for plugging external tools into AI agents. An MCP server exposes custom tools the agent can call.
Monorepo
A single git repo holding many related projects (multiple services, multiple packages, sometimes multiple platforms).
Path glob
A pattern that matches file paths. ios/** matches everything under ios/ at any depth. Used in skills' paths: field.
Plugin
An installable bundle of skills, hooks, and MCP servers. Lets a team distribute its AI Layer in one command.
Progressive disclosure
The design principle that a skill's main file is short, with deeper details in linked reference files that load only when needed.
RAG
Retrieval-Augmented Generation. An older approach where the codebase is indexed and chunks are retrieved at query time. Doesn't scale well in actively-developed codebases.
Repository (repo)
A folder under version control (almost always git). The unit Claude Code operates on.
Session
One continuous conversation with the agent, from claude until you close the terminal or run /clear.
Skill
A folder with a SKILL.md file declaring reusable expertise. Loads only when relevant — scoped via the paths: field.
Standard input / output / error
The three default streams a program can read from or write to. Hooks receive a payload on standard input and inject context by writing to standard output.
Subagent
A separate Claude Code instance launched by the main agent. Has its own context window. Used to split exploration from editing.
Symbol
A named code element — a function, class, variable, constant. LSP-based search finds symbols precisely; grep finds text that may or may not be a symbol.
YAML frontmatter
See Frontmatter. YAML is the syntax used inside the --- block.
